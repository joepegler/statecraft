## Why teams use Statecraft

- Reuse setup across tests instead of rewriting chain and wallet bootstrapping per file.
- Keep tests deterministic with pinned forks (`blockNumber`) and explicit fixture ordering.
- Choose managed per-scenario lifecycle or external suite-scoped lifecycle when runtime startup cost matters.
- Keep TypeScript context accurate as steps compose (`publicClient`, `walletClient`, `wallet`, deployments).
- Keep test bodies focused on behavior, not harness wiring.

## Use Statecraft when

- your tests are already in `vitest` and you want Ethereum setup to be explicit and reusable
- you need repeatable fork-based tests with a pinned block
- you want fast test setup for wallets, ERC-20 balances, and contracts without hiding setup in globals

## Do not use Statecraft when

- you are looking for a full replacement for Foundry or Hardhat
- your testing workflow is not Vitest + viem based
- you need production token issuance semantics rather than test-only state seeding helpers

<Callout type="info">
  Statecraft is intentionally focused on TypeScript + Vitest + viem integration workflows. Use it
  alongside Foundry or Hardhat where those tools are a better fit.
</Callout>

## Minimal example

```ts
import { test, expect } from "vitest";
import { parseEther } from "viem";
import { scenario, withChain, withFundedWallet } from "@statecraft/vitest";

test(
  "funded wallet on local chain",
  scenario(
    withChain(),
    withFundedWallet({ balance: parseEther("1") }),
    async ({ wallet, publicClient }) => {
      const balance = await publicClient.getBalance({ address: wallet });
      expect(balance).toBe(parseEther("1"));
    },
  ),
);
```

For richer scenarios (forks, ERC-20, bytecode injection, deployments), see `packages/examples/examples/scenarios.test.ts`.
